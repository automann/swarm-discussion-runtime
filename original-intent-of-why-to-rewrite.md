The primary use case for the `swarm-discussion` skill is: when a user is collaborating with a parent agent and encounters complex discussion requirements, they invoke this skill to summon a group of experts to discuss outside the current context window. After the discussion concludes, the parent agent makes decisions or executes the next steps based on the discussion results (including conclusions, artifacts, and the discussion process).

Several major issues are currently being faced in the implementation:  

1. The parent agent bears too much responsibility, both to coordinate and manage the process of the entire discussion and to handle specific messages and tool calls. This leads to the parent agent's context window being overly occupied and pollutes the parent agent's context space, without fully utilizing the advantages of multi-agent parallel execution. Additionally, the current context information sent to the child agent is overly verbose, containing a large amount of discussion history and detailed information, some of which is not necessary for the child agent's current task, thereby increasing the complexity and burden on the child agent. This ultimately results in even lightweight discussions requiring longer times and more token overhead to complete.  

2. The current context sent to the child agent does not effectively include a summary of the parent agent's context (background of the issue, intent, and main objectives). This leads to the child agent potentially experiencing intent drift during the discussion, failing to focus on the parent agent's core requirements, ultimately resulting in suboptimal discussion efficiency and quality.
Possible solutions are: 1. First, since we are already using a custom agent, we can fully grant the sub-agent (custom agent) more autonomy and responsibility. I believe we can grant it permission to use the 6 tools: read / bash / edit / write / glob / grep, allowing it to independently collect, analyze, and process information without excessive intervention from the parent agent.

2. Once the sub-agent has more autonomy and responsibility, we can further break down the current responsibilities of the parent agent into different custom agents, each playing a distinct role.

3. When designing the context information sent to the sub-agent, we need to ensure it includes a summary of the parent agent's context, especially the problem background, intent, and main objectives. This will help the sub-agent better understand the parent agent's core requirements during discussions and avoid shifts in intent. Additionally, we need to design a mechanism to ensure the sub-agent continuously aligns with the parent agent's core requirements during discussions. For example, we can design a periodic alignment mechanism, allowing the sub-agent to review and align with the parent agent's core requirements at different stages of the discussion, ensuring the discussion always revolves around the parent agent's core requirements. A possible solution is: the plugin internally maintains a context-generator.md (placed in the same directory as persona-generator.md). When the parent agent calls the swarm-discussion skill, the parent agent generates a context-summary.md based on this template and the current context information, then lets the sub-agent read this document as part of its own context during initialization. The context-summary.md should include the background of the current problem, intent, main objectives, and some key information and data, helping the sub-agent better understand the parent agent's core requirements and continuously align with them during discussions.
