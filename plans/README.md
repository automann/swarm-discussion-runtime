## Intent-alignment context (why these six)

The audit compared the repo against the founding problem statement. Verdicts:
child-context verbosity (P2) and parent-intent injection (P3) are achieved;
the tool-autonomy proposal (S1) was deliberately and reasonably superseded by
the capability-profile design; multi-role orchestration (S2) is deferred with
documented reasons (`docs/FUTURE-EXECUTORS.md`). The remaining gaps are what
these plans close: the runtime's own stdout undermines the parent-context
goal (001), the runtime still forces parent JSON assembly its own
architecture forbids (002), the founding cost complaint is unmeasurable
(004), and the founding doc's one named deliverable — a context-generator
template — is missing (005). 003 and 006 close drift channels found during
the same audit.

## Findings considered and rejected

- **Certify the legacy fixtures end-to-end**: verified to FAIL by
  construction — `fixtures/legacy/*` predate v2 artifacts (no transport/,
  prompts/, events.jsonl), so adapter-smoke and validate-loop correctly
  reject them. The real certification target is the first adapter's live
  discussion.
- **Readonly-profile exercise harness**: already covered —
  `fixtures/e2e/minimal-v2` + `tests/test_phase4_trace_evidence.py` +
  `tests/test_phase6_capability_profiles.py` exercise expert-readonly
  citability end-to-end. The remaining gap (a host actually granting tools)
  is adapter-side and explicitly deferred in `docs/FUTURE-EXECUTORS.md`.
- **`append_event` O(n²) seq scan / per-checkpoint progress.md re-read**:
  negligible at realistic discussion scale (≤ a few hundred events);
  optimize only if events.jsonl ever exceeds ~10k lines.
- **Composed `round-step` command**: command granularity is intentional
  (responsibility split in `ARCHITECTURE.md`); `nextHelperCommand` already
  guides sequencing. Revisit only if adapter authors report sequencing
  burden.
- **`roundId`/`round` duality and `agentId`/`agent_id` aliases**: harmless
  host-compat; document in `docs/ADAPTER-SPEC.md` when an adapter actually
  hits it.
- **`persona-plan` command**: persona generation requires LLM judgment the
  runtime cannot own; at most a persona-JSON validator later, after an
  adapter shows the need.
- **Advisory alignment-check / drift-score phase** (founding "periodic
  alignment"): deferred, not rejected — context summaries are already
  re-injected every phase, and there is no evidence yet (legacy fixtures
  stayed on topic) that drift detection is needed. Revisit with data from
  the first adapter's real discussions.
- **Old-vs-new cost/quality benchmark**: blocked on the first certified
  adapter; the runtime-side prerequisite is plan 004. Record as a Phase 7
  acceptance item when the adapter milestone lands (noted in 004's
  maintenance notes).
