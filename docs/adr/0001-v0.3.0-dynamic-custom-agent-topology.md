- **Status:** Accepted (2026-06-21)

## Context

Through v0.2.x the two host adapters diverged in execution shape:

- **Claude** ran a *nested sub-agent* orchestrator (depth ≥3 verified) that spawned a **static** plugin persona agent (`swarm-discussion:swarm-expert`); persona identity lived in the runtime-built prompt.
- **Codex** could not nest (`agents.max_depth = 1`), so it used a *dedicated coordinator thread* and, until now, ordinary independent expert threads.

The maintainer's intent for v0.3.0 is **one topology on both hosts**, with **dynamic, topic-derived, project-scoped custom expert agents**:

```
parent session
  → creates dynamic custom expert agents for THIS topic (project-scoped files)
  → creates ONE coordinator background session
  → coordinator spawns the custom expert subagents, distributes context,
    aggregates results, returns a compact synthesis index
  ← parent relays synthesis, then terminates the coordinator and deletes the
    dynamic agent files
```

Two host facts make this shape **necessary**, not stylistic:

1. **Custom agents load at session start.** Claude: *"Subagents are loaded at session start. If you add or edit a subagent file directly on disk, restart your session to load it."* Codex Probe 1: writing `.codex/agents/*.toml` and spawning it **in the same running thread** fails with `unknown agent_type`. Therefore a dynamically-created custom agent is only visible to a **freshly-started session** (Claude background session / Codex coordinator thread), proven by Codex Probe 2.
2. **The parent must stay thin and alive.** A fresh *foreground* session would block the parent; a *nested* sub-agent on Claude cannot see the just-written files. The resolution on both hosts is a **detached coordinator session** the parent dispatches, polls, and tears down.

So the dynamic-custom-agent requirement *forces* the coordinator-background-session, and the background-session keeps the parent thin — the topology is internally coherent.

A Claude review of the Codex design surfaced a gap (the "provenance decision"): the Codex coordinator wants runtime artifacts to record *which projected custom agent produced which payload* (projected name, file hash, host id, invocation form), but the runtime transport artifacts have no home for that — `runtime/swarm/transport.py:_validate_spawn_order` normalizes each spawn-order entry down to `{agentId, persona, token?}` and drops the rest. This ADR resolves that.

## Decision

### D1 — Unified topology (both hosts)

Adopt `parent → coordinator background session → dynamic custom expert subagents` as the v0.3.0 shape for **both** adapters. The runtime stays host-agnostic and host-neutral; each adapter maps the shape to its host primitives.

```
PARENT (thin, stays in the user's session)
  1. collect brief
  2. derive per-topic expert agent files from a template + runtime persona plan
       Claude: .claude/agents/swarm-<runId>-<role>.md
       Codex : .codex/agents/swarm-<runId>-<role>.toml
  3. dispatch ONE coordinator background session
       Claude: claude --bg --agent swarm-coordinator --name swarm-<runId> "<packet>"
       Codex : create one dedicated coordinator thread with the coordinator contract
  4. poll for the terminal index; relay only synthesis + artifact paths
  5. terminate the coordinator + delete the run-scoped agent files it created

COORDINATOR (background session / dedicated thread — fresh, sees the new agents)
  runtime init → context-build → prompt-build (per expert)
  → spawn each projected expert (depth-1 subagents) with the runtime prompt only
  → persist raw host wait batches via transport-append-batch
  → transport-collect → append-message → checkpoint → finalize-round
  → validate-host-step / validate-loop / validate-discussion → trace → evidence
  → return a compact terminal index

EXPERT (no-tools leaf custom agent)
  receive ONE runtime-produced prompt → return ONLY the requested JSON → stop
```

The dynamic custom-agent file is a thin **identity/visibility shell**; the per-phase **content** still comes from the runtime `prompt-build` artifact (see ADAPTER-SPEC entry contract). This preserves "runtime owns prompt construction" while giving each expert a distinct, topic-named host identity.

### D2 — Claude coordinator is a background session (hard dependency)

The Claude adapter depends on Claude Code **background sessions** (`claude --bg`, `claude agents --json`, `claude logs|stop|rm|respawn`), the only mechanism that gives a fresh, detached coordinator session able to load dynamically-written `.claude/agents/`. This is accepted as a hard dependency despite the feature's *research-preview* status (see Risks R1). No nested-orchestrator fallback ships in v0.3.0.

### D3 — Provenance is runtime-owned via a host-agnostic `agentDescriptor`

The runtime is the cross-host source of truth, and adapters must not create a second transport store. Therefore projected-custom-agent provenance becomes **runtime-owned**, carried by an optional, host-agnostic `agentDescriptor` on each spawn-order entry, summarized in `host-step.json`, and echoed beside each `collect-result.json` result. (Answers the handoff: Q1 = Option 1; rejects the adapter-sidecar and claim-downgrade options.)

#### `agentDescriptor` contract

Per **spawn-order** entry (all `agentDescriptor` fields optional, but see the projected-gate below):

```jsonc
{
  "agentId": "<host id: Codex agent_id; Claude subagent name/id>",
  "persona": "<role/persona label>",
  "token": "<optional fallback-match token>",
  "agentDescriptor": {
    "projectedName":   "swarm-<runId>-<roleSlug>",          // dynamic custom-agent name
    "projectedPath":   ".claude/agents/swarm-<runId>-<roleSlug>.md", // or .codex/agents/...toml
    "projectedSha256": "<sha256 of the projected agent file at spawn time>",
    "agentType":       "<host agent type used to spawn — normally projectedName>",
    "invocationForm":  "explicit_spawn" | "at_mention",
    "promptRef":       "prompts/r001/<phase>/<agent-id>/prompt.txt"
  }
}
```

`host-step.json` gains a host-agnostic summary under the existing
`transport` object (which already allows additional properties):

```jsonc
"transport": {
  "spawnPrimitive": "...", "waitPrimitive": "...", "resultKey": "agent_id|name",
  "customAgentProjection": {
    "projected": true,
    "agentSourceDir": ".claude/agents" | ".codex/agents",
    "count": <n>
  }
}
```

`collect-result.json` results carry the descriptor so certification can attribute every payload:

```jsonc
"results": [
  { "persona": "...", "agentId": "...", "agentDescriptor": { ... }, "result": { ... } }
]
```

**Naming:** host-agnostic `agentDescriptor` (not Codex-specific `customAgent`). **Schema version:** stays `schemaVersion: 1` — the change is purely additive/optional; it bumps to `2` only if any field becomes *required* at the schema level.

#### Invocation form (handoff Q3)

Certification requires a host id usable by `transport-collect`, so the coordinator **defaults to explicit spawn** and records `invocationForm: "explicit_spawn"`. `@<agent-name>` is allowed only as a convenience and recorded as `invocationForm: "at_mention"`, but never replaces the captured host id.

#### What certification proves (handoff Q4)

**Runtime-owned blockers** (gate certification):
- every `collect-result` result links to an `agentDescriptor` with a non-empty `projectedName`, a well-formed `projectedSha256`, and a `promptRef` that resolves to an existing prompt artifact;
- when `host-step.transport.customAgentProjection.projected == true`, *every* spawn-order entry carries a descriptor;
- every `projectedName` is **run-scoped** — it embeds the discussion/run id recorded in the projection manifest — so a file leaked by one run can never be silently reused by another (structural anti-contamination, not just "please clean up");
- a **projection manifest** (`projection-manifest.json` in the discussion tree) is present, well-formed, its entries match the descriptors, and it records a `deletionStatus`.

This proves fan-out was attributable to named, run-scoped projected agents driven by runtime prompts — not simulated, and not contaminated by a prior run.

**Cleanup is a real blocker, split by what each layer can see.** The runtime validates the projection manifest's *shape* and that `deletionStatus` is present (it cannot see the host's `.claude/agents` / `.codex/agents` directory). The **adapter release gate** hard-blocks on actual zero-residue: a `find`-based check proving no run-scoped projected file survives after success, failure, and timeout paths. An adapter release is not certified until **both** pass.

**Host truth** (that the host actually invoked that custom agent, and that the files were actually deleted) is proven only by the retained real-host smoke plus the adapter residue check, not by the runtime alone — stated honestly in ADAPTER-SPEC (Risk R6).

### D4 — Projection-required certification for v0.3.0 (no old-path loophole)

The plan-008 gate is *opt-in*: it fires only when a discussion declares projection. On its own that would let an adapter certify the **old** spawn path while its docs claim the projected topology. To close the loophole, **a v0.3.0 adapter release MUST certify a *projected* discussion**: certification runs in a projection-required mode (`certify_adapter.py --require-projection`) that fails unless the discussion declares `customAgentProjection.projected == true`, every result carries a descriptor, and the projection manifest is present and consistent. Until an adapter's retained smoke passes projection-required certification, its docs/contracts MUST describe the projected topology as **probe-only evidence**, not product truth. (This is the binding form of handoff Q5.)

## Consequences

**Positive**
- One mental model and one certification target across hosts; the runtime contract — not cross-agent code review — keeps them consistent.
- Per-topic expert identities (better host transcripts/observability) without moving prompt construction out of the runtime.
- Provenance is first-class and machine-checkable; "did the adapter really fan out to projected experts?" becomes a runtime gate, closing the Codex P1.

**Negative / costs**
- Claude takes a hard dependency on a research-preview feature (R1).
- Each discussion now spends N expert subagents **+** one coordinator background session = N+1 independent contexts/quota on Claude (R7).
- New lifecycle surface to get right: dynamic-file projection, collision-safe naming, and guaranteed cleanup on every terminal/failure path (R3).

## Alternatives considered

| Option | Why not |
|---|---|
| **Claude nested-orchestrator (keep v0.2.x)** | A nested sub-agent cannot see `.claude/agents/` written after the parent session started, so it cannot use dynamic file-based custom agents. Incompatible with the intent. |
| **Session-scoped `--agents` JSON** instead of files | Avoids file projection/cleanup, but the maintainer wants project-scoped *files* (symmetry with Codex `.codex/agents/`), and it still requires a fresh session — i.e. still a background session. Noted as a possible future simplification, not chosen. |
| **Adapter-owned provenance sidecar** | Creates a second transport store; violates "runtime is the source of truth" and breaks cross-host consistency. Rejected (handoff Q1 Option 2). |
| **Downgrade Codex claims, no runtime change** | Leaves "projected fan-out" uncertifiable; blocks the whole intent. Rejected (handoff Q1 Option 3). |
| **Background-session + nested fallback (hedge)** | Considered and explicitly declined by the maintainer in favor of full symmetry (D2). |

## Risks & mitigations

- **R1 — Claude background sessions are research-preview** (interface may change; local-only; dies on machine shutdown). *Mitigation:* the adapter `doctor` probes background-session availability and fails fast with a clear capability error; pin the minimum Claude Code version in the adapter README; treat this as the headline release risk in the adapter's notes.
- **R2 — Worktree auto-isolation.** A Claude background session moves into `.claude/worktrees/` on first file write, so coordinator artifacts would land off the main checkout. *Mitigation:* set `worktree.bgIsolation: "none"` in the adapter's project settings, or have the coordinator write to an absolute `discussionDir` the parent passes; assert the resolved `discussionDir` in the terminal index.
- **R3 — Dynamic-file lifecycle & cross-run contamination.** Stale projected agent files are a real hazard, not just untidiness: because a newly-created coordinator session loads whatever `.claude/agents` / `.codex/agents` files exist, a file leaked by run A could be picked up by run B's coordinator. Layered mitigation: (a) **run-scoped names** (`swarm-<runId>-<role>`, runId embedded) make collision/reuse structurally impossible and are runtime-validated against the projection manifest; (b) a **projection manifest** records created paths + sha256 + `deletionStatus`; (c) the **adapter release gate** hard-blocks on a zero-residue `find` check across success/failure/timeout paths. The parent deletes only the exact files it created and never scans/deletes user or plugin agents. (See D4 and the Q4 cleanup split.)
- **R6 — Runtime proves consistency, not host truth.** The runtime cannot verify the host actually invoked the named agent; only the retained real-host smoke does. Documented in ADAPTER-SPEC.
- **R7 — Cost.** N+1 contexts per discussion on Claude; document expected expert counts and that background sessions consume quota independently.

## Per-repo rollout (v0.3.0)

Execute **runtime-first**; do not re-vendor adapters until the runtime artifacts/tests land.

1. **runtime** — `plans/007` (agentDescriptor in transport/collect/schema/validation + tests) → `plans/008` (projected-fan-out certification + fixture + ADAPTER-SPEC/HOST-ADAPTERS/RUNTIME-PACKAGE-BOUNDARY topology docs). Tag a runtime commit for vendoring.
2. **swarm-discussion-claude** — implement the parent projection (+ write the projection manifest) + `claude --bg` coordinator lifecycle + coordinator/expert agents + `doctor` capability probe; re-vendor the runtime; drive & retain a real projected smoke; certify with `--require-projection` **and** pass the zero-residue cleanup check; tag `v0.3.0`. Keep topology claims probe-only until this passes.
3. **swarm-discussion-codex** — align the existing PRD to the final `agentDescriptor` + projection-manifest shape; re-vendor; retain a new projected smoke; certify with `--require-projection` + zero-residue check; tag `v0.3.0`. Keep current claims probe-only until this passes.
4. **swarm-discussion** — bump both marketplace pins to the certified `v0.3.0` adapter releases; CHANGELOG. No structural change.

## Review incorporated (2026-06-21, Codex adversarial review)

A Codex adversarial review flagged two gaps, now addressed:

- **[high] Opt-in gate could let an adapter certify the old spawn path while claiming the projected topology.** Added **D4 — projection-required certification** (`--require-projection`) and the binding rule that adapter topology claims stay *probe-only* until projection-required certification passes.
- **[medium] Cleanup was too soft given the cross-run contamination hazard.** Elevated cleanup to a layered, enforced blocker: run-scoped `projectedName`s validated against a runtime-owned **projection manifest** (Q4), plus an **adapter zero-residue release gate** across success/failure/timeout (R3). The runtime can't see the host's agent directory, so the guarantee is split — runtime validates manifest shape + run-scoped naming; the adapter proves actual deletion.

Plans 007 (run-scoped name validation + projection-manifest schema), 008 (`--require-projection` mode + manifest validation + negative tests), and the Claude adapter plan 001 (manifest writing + zero-residue release gate) were updated to match.
